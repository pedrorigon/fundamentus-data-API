# Releasing

This project uses semantic versioning. Tags use the `vMAJOR.MINOR.PATCH` format.

GitHub Releases are created automatically by `.github/workflows/release.yml` when a matching tag is pushed.

## Version Policy

- Patch: bug fixes, documentation corrections and dependency updates that do not change the public API contract.
- Minor: backwards-compatible endpoints, response fields, parser support or configuration.
- Major: breaking endpoint, response, parser, cache or configuration changes.

Until `1.0.0`, minor versions may still include breaking changes when parser behavior or data contracts need to mature. Breaking changes must be called out clearly in `CHANGELOG.md`.

## Changelog Automation

`CHANGELOG.md` is generated by [git-cliff](https://git-cliff.org) from conventional commit messages (`feat:`, `fix:`, `perf:`, ...) using the configuration in `cliff.toml`. Do not edit it by hand:

- `.github/workflows/changelog.yml` regenerates the `[Unreleased]` section on every push to `main` and commits the result automatically.
- `.github/workflows/release.yml` generates the release notes for the pushed tag directly from the commit history and syncs the new version section back to `main` after publishing the release.

Changelog quality therefore depends on commit message quality: write conventional commit subjects that describe the user-facing change. `docs`, `style`, `test`, `chore`, `ci` and `build` commits are excluded from the changelog.

## Version Locations

The release workflow injects the pushed tag version into `pyproject.toml` before building artifacts. Runtime version is resolved from installed package metadata, with `FUNDAMENTUS_API_VERSION` available as an explicit environment override.

`app/main.py` and `/health` both expose the runtime version from `app.__version__`.

## Release Checklist

1. Optional: update the local development version in `pyproject.toml` when you want local package metadata to show the new version before tagging. This is not required for release artifacts.
2. Run the full validation suite:

```bash
uv run ruff format --check .
uv run ruff check .
uv run mypy app
uv run pytest
```

3. Optional: build the Python distribution and Docker image locally:

```bash
uv build
docker build -t fundamentus-data-api .
```

4. Create and push the signed or annotated tag:

```bash
git tag -a vX.Y.Z -m "vX.Y.Z"
git push origin main --tags
```

5. Check the generated GitHub Release and artifacts. The release workflow generates the release notes from the commit history and commits the new `CHANGELOG.md` section back to `main` automatically.

## Release Pipeline

The release workflow:

- Runs formatting, linting, type checking and tests.
- Reads the version from the pushed tag.
- Injects the tag version into `pyproject.toml` for the release build.
- Generates the release notes from the commit history with git-cliff.
- Builds the Python wheel and source distribution with `uv build`.
- Builds a Docker image and stores it as a `.docker.tar` artifact.
- Generates `SHA256SUMS`.
- Creates a GitHub Release with all files in `dist/` attached.
- Regenerates `CHANGELOG.md` on `main` with the new release section.
